# Cudy TR3000 养老院跌倒检测实施方案

> 整理时间：2026-03-08
> 设备：Cudy TR3000（256MB 闪存版）
> 目标：养老院楼道老人跌倒检测 + 护工通知

---

## 一、设备分析：Cudy TR3000

### 1.1 硬件规格

| 参数 | 详情 |
|------|------|
| **SoC** | MediaTek MT7981B (Filogic 820) |
| **CPU** | 双核 ARM Cortex-A53 @ 1.3GHz |
| **RAM** | 512MB DDR4 |
| **闪存** | 256MB SPI NAND |
| **WiFi 芯片** | MediaTek MT7976CN |
| **WiFi 标准** | 802.11ax (Wi-Fi 6) |
| **频段** | 2.4GHz (574Mbps) + 5GHz (2402Mbps) |
| **MIMO** | 2.4GHz: 2×2:2 / 5GHz: 2×3:2 |
| **天线** | 4 根内置天线（U.FL 接头） |
| **以太网** | 1× 2.5Gbps WAN + 1× 1Gbps LAN |
| **USB** | USB 3.0 |
| **功耗** | 5V / 3A (15W) |
| **尺寸** | 口袋大小便携路由器 |
| **当前系统** | OpenWrt（已刷机） |

### 1.2 CSI 提取能力评估

**结论：可行，MT7981B/MT7976CN 已被社区验证支持 CSI 提取。**

关键证据：
1. **MtkCSIdump 项目**（[GitHub](https://github.com/MtkWifiRev/MtkCSIdump)）已实现 MediaTek 芯片的 CSI 提取，通过 mt76 驱动补丁实现，不需要修改固件
2. **ekstra-csi 项目**（[OpenWrt 论坛](https://forum.openwrt.org/t/ekstra-csi-full-metadata-csi-extraction-for-mt76-on-openwrt/247100)）是改进版，保留完整元数据（源 MAC、RSSI、SNR、包序列号）
3. MT7981B 使用的 mt76 驱动在 OpenWrt 中已有成熟支持
4. OpenWrt One（同样使用 MT7981B + MT7976C）已被验证可以进行 CSI 采集

### 1.3 与项目原生硬件对比

| 特性 | ESP32-S3（项目推荐） | Cudy TR3000（你的设备） |
|------|---------------------|----------------------|
| CSI 支持 | 原生支持，乐鑫官方 API | 通过 mt76 补丁支持 |
| CPU 算力 | 单核 240MHz | 双核 1.3GHz（约 **5x** 更强） |
| 内存 | 512KB SRAM + 8MB PSRAM | 512MB DDR4（约 **60x** 更多） |
| 存储 | 4-16MB Flash | 256MB NAND（约 **16x** 更多） |
| WiFi 标准 | WiFi 4 (802.11n) | WiFi 6 (802.11ax) |
| 天线 | 1-2 根 | 4 根 |
| MIMO | 1×1 | 2×2 / 2×3 |
| 子载波数 | 52-56 | 56+ (HE模式更多) |
| 操作系统 | FreeRTOS / 裸机 | Linux (OpenWrt) |
| 运行 Rust | 有限支持 (no_std) | 完整 Linux Rust 工具链 |
| 价格 | ~$8 | ~$55（你已有） |
| 功耗 | ~0.5W | ~15W |
| 网络功能 | 需外接路由器 | **本身就是路由器** |

**关键优势**：TR3000 算力远超 ESP32，可以直接在设备上运行完整的信号处理 + 推理管线，而不需要外接服务器。同时它本身就是路由器，既是 WiFi 信号发射源又是 CSI 采集器，**一台设备同时扮演发射端和接收端的角色**。

### 1.4 风险与限制

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| CSI 补丁兼容性 | mt76 补丁可能需要针对 TR3000 调整 | 先在测试环境验证，保留原始固件备份 |
| 单设备覆盖范围 | 一台路由器只有一个视角 | 楼道场景（狭长空间）单设备可能够用 |
| 功耗较高 | 15W vs ESP32 的 0.5W | 养老院有供电，不是问题 |
| 散热 | 持续运行可能发热 | 安装在通风位置 |
| 闪存寿命 | NAND 有写入次数限制 | 日志和临时数据放 tmpfs |

## 二、养老院楼道部署方案

### 2.1 场景分析

```
养老院楼道特点：
- 狭长空间（通常 2-3 米宽，10-30 米长）
- 两侧有房门
- 可能有拐角
- 需要 24/7 不间断监控
- 老人可能独自行走
- 关键需求：检测跌倒 → 通知护工

隐私要求：
- 不能使用摄像头（失能老人可能衣着不全）
- WiFi 信号感知是理想方案
```

### 2.2 系统架构

```
┌─────────────────────────────────────────────────────┐
│                    养老院楼道                          │
│                                                     │
│  [Cudy TR3000]──────WiFi 信号────────→ 楼道空间      │
│       │                                             │
│       │ CSI 数据采集                                  │
│       ↓                                             │
│  ┌──────────────┐                                   │
│  │ 信号处理引擎   │  ← 运行在 TR3000 上               │
│  │ (Rust 编译)   │                                   │
│  │              │                                   │
│  │ 1. CSI 提取  │                                   │
│  │ 2. 降噪滤波  │                                   │
│  │ 3. 特征提取  │                                   │
│  │ 4. 跌倒判定  │                                   │
│  └──────┬───────┘                                   │
│         │                                           │
│         │ 检测到跌倒                                  │
│         ↓                                           │
│  ┌──────────────┐     ┌──────────────┐              │
│  │  告警服务     │────→│  护工手机/    │              │
│  │ (HTTP/MQTT)  │     │  护士站终端   │              │
│  └──────────────┘     └──────────────┘              │
└─────────────────────────────────────────────────────┘
```

### 2.3 单设备 vs 多设备方案

#### 方案 A：单台 TR3000（最简方案）

```
适用场景：10-15 米直线楼道
优点：成本最低，部署最简单
缺点：覆盖范围有限，只有一个观测角度

工作模式：
- TR3000 作为 AP 发射 WiFi 信号
- 老人手机/手环 或 楼道内其他 WiFi 设备作为反射源
- TR3000 自身采集 CSI 变化
```

#### 方案 B：TR3000 + 1-2 个低成本 WiFi 终端（推荐）

```
适用场景：15-30 米楼道
成本：TR3000 + 2× ESP32 (~$16) ≈ $71

工作模式：
- TR3000 作为 AP + 中心处理器
- 2 个 ESP32 作为 WiFi Station 持续发包
- TR3000 采集来自 ESP32 的 CSI 数据
- 多条信号路径覆盖整个楼道

部署示意：
  [ESP32-A]────────────[TR3000]────────────[ESP32-B]
  楼道一端              楼道中间              楼道另一端
         ↘              ↕               ↙
          ──── 信号路径覆盖整个楼道 ────
```

#### 方案 C：多台 TR3000 组网（大面积覆盖）

```
适用场景：整层楼多条楼道
每台 TR3000 覆盖一段楼道
通过有线/无线 Mesh 互联
统一管理和告警
```

**推荐方案 B**：成本低、覆盖好、可靠性高。

```

│  └───────────────────────────────────────────────┘   │
```

## 三、实施步骤（分阶段）

### 阶段 1：环境准备与 CSI 验证（1-2 周）

**目标**：确认 TR3000 能稳定采集 CSI 数据

```
步骤：
1. 备份当前 OpenWrt 固件（防止变砖）
   → `dd if=/dev/mtd0 of=/tmp/firmware-backup.bin`
   → 通过 SCP 拷贝备份到电脑

2. 编译带 CSI 补丁的 OpenWrt 固件
   → 基于当前 OpenWrt 版本
   → 应用 MtkCSIdump 或 ekstra-csi 的 mt76 补丁
   → 交叉编译 aarch64 目标

3. 刷入新固件并验证 CSI 采集
   → 使用一台手机作为 WiFi 客户端
   → 在手机附近走动
   → 观察 CSI 数据变化是否能反映人体运动

4. 采集基线数据
   → 空楼道（无人）的 CSI 基线
   → 有人正常行走的 CSI 数据
   → 模拟跌倒动作的 CSI 数据
```

**交付物**：一份 CSI 数据采集报告，确认数据质量

### 阶段 2：信号处理管线移植（2-3 周）

**目标**：将 RuView 核心模块交叉编译到 TR3000 上运行

```
步骤：
1. 搭建 Rust 交叉编译环境
   → target: aarch64-unknown-linux-musl
   → 针对 Cortex-A53 优化

2. 选择并编译核心 crate
   → wifi-densepose-core
   → wifi-densepose-signal（可能需要裁剪不需要的模块）
   → wifi-densepose-vitals
   → wifi-densepose-config

3. 开发 CSI 适配层
   → 将 mt76 CSI 数据格式转换为 RuView 的 CsiFrame 格式
   → 适配 MediaTek CSI 与 ESP32 CSI 的差异

4. 在 TR3000 上运行并验证
   → 实时处理 CSI → 输出信号特征
   → 验证 CPU/内存占用在合理范围内
```

**交付物**：可在 TR3000 上运行的信号处理二进制文件

### 阶段 3：跌倒检测算法开发（2-3 周）

**目标**：实现可靠的跌倒检测

```
跌倒检测原理：
- 跌倒事件在 CSI 中表现为：
  1. 突然的大幅信号变化（身体快速位移）
  2. 随后信号趋于静止（倒地不动）
  3. 这个模式与正常活动（走路、弯腰、坐下）有显著差异

算法方案（由简到复杂）：

方案 A - 规则阈值法（快速原型）：
  → CSI 方差突变检测
  → 运动能量带通滤波
  → 静止时间窗口判断
  → 准确率预期：80-85%

方案 B - 机器学习法（推荐）：
  → 在 TR3000 上运行轻量级模型
  → 特征：CSI 统计量 + 频域特征
  → 模型：轻量级 CNN-LSTM 或逻辑回归
  → 准确率预期：90-95%
  → 参考：项目已有 ADR-048 自适应分类器

方案 C - 姿态估计法（最精准）：
  → 完整的 17 关键点姿态重建
  → 基于姿态变化判断跌倒
  → 需要更多算力和训练数据
  → 准确率预期：95%+
```

**推荐从方案 A 快速验证概念，然后迭代到方案 B。**

### 阶段 4：告警系统开发（1-2 周）

**目标**：检测到跌倒后及时通知护工

```
告警方案：

1. 局域网方案（最简单，推荐先实现）：
   → TR3000 运行轻量级 HTTP 服务
   → 检测到跌倒 → 发送 HTTP 请求到护士站电脑
   → 护士站电脑弹窗 + 声音警报
   → 显示跌倒位置（哪台设备/哪段楼道）

2. 手机推送方案：
   → 通过 MQTT 推送到护工 App
   → 或通过企业微信/钉钉 Webhook 发通知
   → 需要养老院有互联网接入

3. 短信/电话方案（紧急备份）：
   → 通过云端短信 API（阿里云/腾讯云）
   → 在网络中断时通过 USB 4G 模块发短信
```

### 阶段 5：现场部署与调优（持续）

```
1. 现场安装
   → 楼道天花板或墙壁高处固定 TR3000
   → 安装辅助 ESP32 节点（如使用方案 B）
   → 接通电源和网线

2. 环境学习
   → 运行 RuView 的房间指纹模型（field_model）
   → 学习楼道的 RF 特征基线
   → 建议学习期 24-48 小时

3. 参数调优
   → 调整跌倒检测灵敏度（降低误报/漏报）
   → 根据实际楼道长度调整覆盖策略
   → 测试不同时段（白天/夜间）的检测效果

4. 持续运维
   → 定期检查设备运行状态
   → 更新检测模型
   → 收集误报/漏报案例用于改进
```

## 四、技术可行性总结

| 维度 | 评估 | 说明 |
|------|------|------|
| **CSI 采集** | ✅ 可行 | MT7981B/MT7976CN 已有社区 CSI 提取方案 |
| **算力** | ✅ 充裕 | 双核 A53 + 512MB RAM 远超 ESP32 |
| **存储** | ✅ 充裕 | 256MB NAND 足够存放程序和模型 |
| **软件生态** | ✅ 成熟 | OpenWrt + Linux 生态完善 |
| **Rust 编译** | ✅ 支持 | aarch64-linux 是 Rust 一级支持目标 |
| **供电** | ✅ 无忧 | 养老院有稳定供电 |
| **覆盖范围** | ⚠️ 需验证 | 单设备覆盖 10-15 米，楼道可能需要多设备 |
| **检测精度** | ⚠️ 需调优 | 取决于环境和算法，需要现场标定 |
| **长期稳定性** | ⚠️ 需观察 | 7×24 运行的散热和内存泄漏需要关注 |
